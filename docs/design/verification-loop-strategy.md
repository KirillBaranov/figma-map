# Verification Loop — стратегические заметки (сессия 2026-07-20)

Статус: черновик для проработки. Не план работ — зафиксированные направления
и решения из брейншторм-сессии. Цель: полный agent loop «макет → верстка →
сходимость к ~100%», где тулза сообщает *что, где и почему* не так, а внешний
агент адаптирует под кодовую базу.

## Статус реализации (обновлено в той же сессии)

Каскад ниже почти целиком реализован. Детальный код-левел план — в
`.claude/plans/parallel-snacking-waterfall.md`; что реально шипнуто — в
[CHANGELOG.md](../../CHANGELOG.md). Кратко по разделам этого дока:

- **«Ядро: единый issue-list»** — сделано (`internal/service/issue.go`,
  `Diff.Issues`, `PixelDiffResult.Issues`), аддитивно поверх старых полей.
- **«Каскад верификации»** — Structured diff (Tier 1) без изменений остаётся
  основным механизмом. Attributed pixeldiff — сделано в объёме: кластеризация
  connected-components + классификация shift/color/other + induced-diff
  subtraction (`internal/render/cluster.go`); полной семантики "missing
  element" из региона нет. VLM — сделано: `reconcile --semantic` теперь
  скоупится на unresolved-кропы вместо всего фрейма
  (`internal/service/reconcile_tier2.go`).
- **«Матчинг нод: инварианты, не ID»** — частично: accessible-text
  (aria-label/alt/title) и component/class-name anchors как tie-break к
  существующему IoU-матчеру (`internal/service/align.go`). Tree-order/
  ancestor-chain сигнал (повторяющиеся карточки) — не сделан, нужна
  валидация на реальных фикстурах с повторами прежде чем шипить.
- **«Семантика: анти-Goodhart»** — `position:absolute` в auto-layout и
  Variable-hint на цветовых диффах сделаны
  (`internal/service/reconcile.go`); hug/fill sizing lint — не сделан, нужен
  новый sizing-mode field через plugin→backend→Go (которого сегодня нигде
  нет), не Go-only правка.
- **«Два слоя истины + гео-дифф»** (кейс слайдера) — сделано целиком:
  `absoluteRenderBounds` от плагина до Go, `verify_reconcile --geoDiff`
  сравнивает его с post-transform DOM-боксом вместо старого молчаливого
  пропуска проверки.
- **«Производительность»** — все 4 пункта из «Приоритет по соотношению
  боль/усилие» реализованы, но каждый в урезанной форме относительно текста
  ниже: dormant-статус выводится из поведения уже идущего запроса (нет
  отдельного ambient-пульса от плагина), job resume работает только на
  быстрый reconnect (не полноценная персистентная очередь с TTL), `get_node`
  стримится и кэшируется, но кэш сбрасывается целиком на любой
  `documentchange`, а не по точному dirty-set. Детали и обоснование урезания
  каждого пункта — в CHANGELOG и в plan-файле.
- **«Стратегия агента: верстка целого макета»** и **«Открытые вопросы»** —
  не тронуты, остаются open questions как есть.

## Принятые рамки (решено в сессии)

- **Тулза — не агент.** Поставляем факты и измерения; интерпретация и правки —
  зона агента. (Совпадает с ADR-0001 dumb-tool.)
- **Матчинг breakpoint-версий макета — ручной.** Агент сам передаёт
  `{frame → breakpoint}`; тулза не связывает версии фреймов между собой.
  Каждый breakpoint — независимая цель верификации со своим issue-list.
- **Никаких атрибутов в клиентском коде.** Ни runtime, ни shadow attrs —
  политическая цена для OSS-адопшена слишком высокая. Матчинг нод — в момент
  диффа, по инвариантам (см. ниже).

## Ядро: единый issue-list вместо голого pixeldiff

Выход верификации — машиночитаемый список в духе линтера:

```
{ node, dom_selector, property, expected, actual, severity, confidence,
  source: structured | pixel | vlm }
```

- Агент чинит топ issue → тулза перепроверяет → список сокращается.
- Критерий остановки становится честным: пустой список выше порога severity,
  а не «pixeldiff на глаз похож».
- `confidence` обязателен — тулза репортит и факт собственной неуверенности
  (особенно по матчингу), чтобы агент не чинил не то.

## Каскад верификации (от дешёвого к дорогому)

### 1. Structured diff — основной механизм
Сравнение двух деревьев: Figma-ноды (ground truth уже есть) vs рендер
(bounding rects + computed styles через extension).

- Закрывает ~70–80% расхождений (spacing, размеры, цвета, типографика)
  точными числами. Детерминированно, бесплатно.
- Главная сложность — матчинг нод, см. раздел про матчинг.

### 2. Attributed pixeldiff — на остаток
Кластеризация diff-пикселей в регионы → проекция на bbox'ы нод →
детерминированная классификация типа дефекта:
- сдвиг (кросс-корреляция даёт точный offset);
- цвет (гистограммы совпадают по форме, не по значениям);
- размер/масштаб;
- отсутствующий/лишний элемент.

Подводные камни: tolerance-полосы на антиалиасинг текста; вычитание
«наведённых» диффов (один сдвиг сверху каскадно красит всё ниже — нашли
сдвиг → виртуально сдвинули → пере-диффнули), иначе 40 issue вместо одного.
Severity = площадь × контрастность региона.

### 3. VLM — последний, опциональный этаж
Только на нерешённые кластеры, только маленькие кропы, только под флагом
(`--vlm`). Задача VLM — *классифицировать* дефект (не тот вес шрифта,
не та иконка, другой кроп картинки), не *измерять* — точные величины
у нас и так есть детерминированно. В критический путь не пускать:
стоимость, латентность, недетерминизм.

## Матчинг нод: инварианты, не ID

Node ID — неправильная валюта: не переживает breakpoint-версии, рефакторинг
макета, copy-paste фрейма. Якоримся на инвариантные свойства:

- текстовое содержимое (текстовые ноды — почти уникальные якоря);
- имя компонента/инстанса;
- роль/тип (кнопка, картинка, инпут);
- относительная позиция в дереве и геометрический порядок.

Алгоритм — класс tree matching как GumTree: сначала уверенные якоря-листья
(совпал текст), затем распространение матчинга вверх/вниз по дереву.
Wrapper-дивы и разная вложенность прощаются.

Известные слабости: повторяющиеся карточки/пункты меню (разруливать
геометрическим порядком), безтекстовые иконки (матчить через родителя +
позицию). Уверенность матчинга — в `confidence` issue.

Опциональный ускоритель: sidecar-манифест `{node → file:line}` как побочный
продукт генерации сниппетов, для проектов, где агент ведёт код с нуля.
Код не трогает, дрейфует при рефакторинге без агента — поэтому опция,
не основа.

## Семантика: анти-Goodhart (чтобы агент не ушёл в position:absolute)

Если метрика — только пиксели, absolute на всё — оптимальная стратегия
агента. Запрет в промпте проигрывает метрике всегда. Семантику встраиваем
в саму метрику:

1. **Issue в валюте layout-интента, не координат.** Figma auto-layout —
   это flex с известными gap/padding/alignment. «Gap 16, в макете 24» →
   дешёвейший фикс `gap: 24px`. «Элемент должен быть на x=312» → дешёвейший
   фикс absolute. Язык отчёта определяет код, который напишет агент.
   Голые x/y — только для реально абсолютных слоёв макета.
2. **Structure lint как часть верификации.** Дешёвые проверки по уже
   собранным computed styles, дающие issue независимо от пикселей:
   - `position: absolute` там, где нода в auto-layout;
   - захардкоженные width/height там, где в макете hug/fill;
   - magic-number hex там, где известна Figma Variable.
   Пиксели сошлись, но lint красный → loop не завершён.
3. **Токены и компоненты в expected.** «expected `--color-primary`
   (#6C5CE7)», «это инстанс `Button/Large` → `<Button size="lg">`»
   (component mapping уже есть).

Граница: lint сравнивает только с тем, что *утверждает макет* (интент
лейаута, токены, компоненты). Мнения о коде (grid vs flex, вложенность) —
зона правил проекта, туда тулзе нельзя.

Бонус: интент-язык в сниппетах снижает дрейф уже на первой генерации —
дешевле всего дифф, который не возник.

## Сложные кейсы: два слоя истины + гео-дифф (кейс слайдера)

Кейс: вертикально-горизонтальный слайдер — declared-значения совпадают,
pixeldiff не сходится. Механика: Figma хранит `relativeTransform` от
верхнего-левого угла родителя, CSS вращает вокруг `transform-origin`
(default 50% 50%). Тот же угол — сдвинутый результат.

1. **Диффить реализованную геометрию, не только стили.** Обе стороны отдают
   post-transform правду: Figma `absoluteRenderBounds` vs DOM
   `getBoundingClientRect`. Ловит ошибку композиции при совпадающих
   declared-значениях и локализует звено: «declared rotate совпадает,
   render bounds смещён на (14, −3) → подозрение на transform-origin».
   Поправки: нормализация DPR/zoom; тени и blur входят в
   `absoluteRenderBounds` (брать bbox без эффектов или сравнивать оба);
   tolerance на текстовые bbox.
2. **Сниппеты переводят трансформации, а не транслитерируют.** Конвертация
   матрицы Figma → CSS (`transform` + правильный `transform-origin` +
   компенсирующий translate) — детерминированная математика, пишется раз.
3. **Лестница эскалации.** Нода не сходится N итераций (видно по истории
   issue-list'ов) → deep dive по ноде: PNG-экспорт именно этой ноды +
   скриншот именно этого DOM-элемента + разложение матрицы + геометрия
   детей. Плотный контекст адресно, вместо всего экрана каждый раз.

## Производительность: большие деревья и живучесть плагина

Проблема: экспорт макета на ~1k слоёв — тяжёлая сериализация; свёрнутая
вкладка Figma троттлит плагин → «соединение потеряно» → после разворота
плагин штормит бэкенд запросами, которые агенту уже не нужны.

Направления (детали — в проработке):

- **Query-driven доступ вместо full-tree экспорта**: поддерево по node id,
  ограничение глубины, проекция полей — агент почти никогда не нуждается
  во всех 1k слоёв разом.
- **Job-протокол с TTL и отменой**: каждый запрос — job с id и сроком
  жизни; после реконнекта плагин спрашивает бэкенд «что ещё актуально»
  вместо replay накопленной очереди. Протухшие jobs истекают — шторм
  устраняется по построению.
- **Резюмируемая чанковая передача**: sequence numbers, идемпотентные
  чанки, докачка с места обрыва вместо рестарта всего экспорта.
- **Дешёвая сериализация**: skip invisible instance children, не читать
  дорогие свойства без запроса (absoluteRenderBounds форсит рендер),
  strip default-значений, дедупликация повторяющихся стилей (style table +
  ссылки — 1k слоёв сильно репетитивны).
- **Кэш + дельты**: экспорт раз, дальше `documentchange`-дельты вместо
  полного пере-экспорта. В verify-loop макет между итерациями не меняется —
  повторные походы в плагин должны отдавать кэш. Dormant-плагин копит дельты
  как dirty-set (не лог событий), иначе шторм вернётся в новой форме.
- **Свёрнутая вкладка — нормальный режим, не сбой.** Плагин — устройство,
  которое периодически спит. Статус трёхзначный: connected / dormant / gone;
  `dormant` — честный ответ тулзы («разверни вкладку»), не ошибка.
  Реконнект = resume сессии по id с последнего подтверждённого seq.
- **Обход с yield'ами** (порции ~50 нод), чтобы не вешать UI-тред вкладки —
  замороженный тред пропускает heartbeat'ы: болезнь сериализации кормит
  болезнь соединения. Реальная стоимость экспорта — не JSON.stringify,
  а переходы через sandbox-границу на каждое чтение свойства (~30k вызовов
  на 1k слоёв × 30 полей).
- **REST API как холодный fallback** для full-tree без живой вкладки.
  Минусы: токен (трение онбординга), rate limits, данные беднее плагинных.
  Опция, не основа.

**Приоритет по соотношению боль/усилие:**
1. TTL/pull-протокол jobs — убивает шторм по построению, маленький.
2. Проекция полей + query-driven — убивает основную стоимость сериализации.
3. Кэш между итерациями loop — умножает выигрыш именно в verify-цикле.
4. Резюмируемые чанки, дельты — полировка после первых трёх.

## Сводка: почему каскад лучше текущего pixeldiff — и лучше ли

Текущее состояние: pixeldiff говорит «что-то не так», но не «что, где и на
сколько». Даже человек с трудом ориентируется по красной простыне; агент —
тем более: он интерпретирует, а не измеряет, и каждая итерация loop — это
догадка.

Что меняет каскад, по классам расхождений:

| Класс дефекта | Pixeldiff сейчас | Каскад |
|---|---|---|
| Spacing/размер/цвет/типографика (~70–80% случаев) | красный регион, агент гадает | точное число + свойство + нода (structured diff) |
| Сдвиг блока | каскадная простыня вниз по экрану | один issue с offset (кластеризация + вычитание наведённых) |
| Ошибка композиции transform (кейс слайдера) | невидим: значения «верны» | гео-дифф render bounds локализует звено |
| Семантика кода (absolute-хак) | невидим: пиксели сошлись | structure lint даёт issue независимо от пикселей |
| «Не та иконка / не тот кроп» | красное пятно без объяснения | VLM-этаж классифицирует (опционально) |

Почему это сработает — механика, а не надежда: агент чинит то, что
сформулировано как незакрытый пункт. «Gap 16 вместо 24» превращается
в правку за одну итерацию; красное пятно — в 2–4 итерации проб. Ожидаемый
эффект — не столько выше финальный процент, сколько **меньше итераций до
сходимости** и предсказуемая остановка (пустой issue-list вместо «на глаз
похоже»).

Честные риски, где может не сработать:
- ошибки матчинга нод → агент чинит не тот элемент (митигируется
  `confidence` + текстовыми якорями);
- residual-класс, который не ловится ни стилями, ни render bounds
  (шрифтовый рендеринг, градиенты) — останется на pixel/VLM этажах;
- Goodhart на новую метрику: агент закрывает issue-list, а картинка хуже —
  поэтому pixeldiff-score остаётся контрольной метрикой второго мнения,
  его не выбрасываем.

**Критерий успеха проверяем на существующем бенче:** итераций до
сходимости и финальный pixel-score с issue-list vs без. Если итераций
не стало меньше — каскад не работает, честно фиксируем.

## Стратегия агента: верстка целого макета (live-кейс «лендинг под 1440»)

Целевой сценарий: есть рабочий код, агенту скармливается целый фрейм
(«адаптируй под 1440px»), он доводит всё сам. Тулза не диктует агенту
процесс, но скилл (figma-map skill) должен описывать рабочий playbook:

1. **Инвентаризация.** Агент запрашивает верхний уровень фрейма (только
   top-level секции, query-driven, без full-tree): список блоков Hero /
   Features / Pricing / Footer с геометрией. Это план работ.
2. **Блок за блоком, не экран целиком.** Для каждой секции: сниппет +
   токены поддерева → маппинг на существующие компоненты кода → правка →
   verify этой секции → закрыть issue-list → следующая. Причины:
   контекст-бюджет агента, локализация issue, ранняя сходимость
   (ошибка в Hero не шумит в диффе Footer).
3. **Существующий код — приоритетная цель маппинга.** Перед генерацией
   нового агент ищет, чем блок уже свёрстан (component mapping + текстовые
   якоря в DOM); правка существующего всегда дешевле и семантичнее
   генерации рядом.
4. **Финальный проход по всему фрейму.** После сходимости всех секций —
   verify целого экрана: ловит межсекционные вещи (вертикальные отступы
   между блоками, общий фон, sticky-элементы).
5. **Эскалация по застрявшим нодам** (deep dive из раздела про сложные
   кейсы) — вместо бесконечных общих итераций.

Роль тулзы в этом playbook: (а) отдавать инвентарь секций дёшево,
(б) verify по произвольному поддереву, не только по фрейму целиком,
(в) issue-list с confidence. Роль скилла: закрепить порядок шагов и правила
проекта (семантика, токены, существующие компоненты). Роль агента: маппинг
на кодовую базу и сами правки.

## Открытые вопросы

- Порог tolerance для текстовых bbox и антиалиасинга — подобрать на бенче.
- Формат issue-list: стабилизировать схему до реализации каскада.
- Deep dive: значение N (итераций до эскалации) и бюджет на ноду.
- VLM-этаж: какая модель, какой бюджет на итерацию, нужен ли вообще
  после каскада 1+2 — решать по остаточным кейсам бенча.
